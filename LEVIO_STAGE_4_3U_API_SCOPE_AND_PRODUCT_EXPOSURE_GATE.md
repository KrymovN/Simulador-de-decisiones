subscriptions.

## Strategic Invariant

Levio is not an AI Chat.

Levio is not an Answer Engine.

Levio is not a generic AI assistant.

Levio is a Decision Simulation Engine.

The immutable target runtime architecture remains:

```text
USER -> SIMULATOR -> DECISION ENGINE -> PROMPT CONTEXT -> AI PROVIDER -> DECISION ENGINE -> SIMULATOR -> UI
```

User Data Controls may expose ownership, export, deletion, consent, retention,
and privacy visibility only around owner-scoped decision simulation artifacts.
They must not expose AI chat history, generic prompt history, assistant
conversation logs, provider logs, embeddings, vectors, memory, or generic
assistant state as Levio product objects.

## Future API Operations Allowed For Exposure

The following operations may be considered for future API exposure after
separate implementation approval:

| Operation | Future API Eligibility | Allowed Initial Shape | Production Caveat |
| --- | --- | --- | --- |
| Export request | Conditionally allowed | Authenticated server-side request that creates or returns a manifest-only export plan | No file, archive, storage object, download URL, or package generation until separately approved |
| Export status | Conditionally allowed | Authenticated status/plan visibility for the resolved owner | Must not expose raw protected payloads or forbidden data categories |
| Deletion request | Conditionally allowed | Authenticated lifecycle-only deletion planning request | No deletion write, hard delete, job, or account deletion orchestration until separately approved |
| Deletion status | Conditionally allowed | Authenticated status visibility for lifecycle-only deletion planning | Must not imply irreversible deletion has executed |
| Consent management | Blocked for write exposure | Read/evaluation-only scope may be considered later | Durable consent ledger, withdrawal workflow, and legal copy require separate approval |
| Retention visibility | Conditionally allowed for visibility only | User-visible category/policy status for owned decision artifacts | Retention jobs, expiration execution, and legal exception handling require separate approval |

## Data That May Leave The Persistence Boundary

Future API responses may expose only owner-scoped, user-facing metadata after
successful server-side auth, canonical principal resolution, and ownership
verification:

- request ID generated by the future server workflow;
- operation type;
- status enum;
- user-facing timestamps;
- owner-scoped resource categories;
- controlled artifact IDs for resources owned by the resolved principal;
- export manifest summary;
- deletion plan summary;
- retention visibility summary;
- consent policy status summary, if legal copy and scope are approved;
- sanitized blockers or user-facing eligibility reasons;
- schema/contract version for client compatibility.

Decision simulation artifact contents may leave the persistence boundary only in
a later separately approved real export package implementation. Stage 4.3U does
not approve payload export.

## Data That Must Never Leave The Persistence Boundary

Future API exposure must never return:

- provider secrets;
- service-role data;
- auth tokens;
- session cookies;
- raw provider references when not needed for user-facing display;
- other-user records;
- internal security signals;
- internal operational logs;
- database rows as raw persistence objects;
- backup contents;
- billing records or subscription internals;
- AI prompts;
- raw AI responses;
- AI provider logs;
- embeddings;
- vectors;
- memory data;
- AI chat history;
- generic prompt history;
- assistant conversation logs;
- generic assistant state;
- legal/internal operator notes.

## Operations That Must Remain Server-Only

These operations must not be performed by the client and must not accept
client-supplied ownership as authorization:

- auth/session reading;
- provider-reference validation;
- canonical `levio_principals.principal_id` resolution;
- ownership verification;
- persistence read provider execution;
- owner-scoped artifact discovery;
- export plan construction;
- deletion plan construction;
- retention evaluation;
- consent evaluation;
- legal-hold or retention-exception checks;
- future export storage/download URL creation, if later approved;
- future deletion lifecycle writes or hard deletion, if later approved.

## Ownership Checks Required Before Exposure

Every future API route must prove all of the following before returning any
owner-scoped data:

- an active authenticated session exists;
- the session resolves to exactly one active registered-user Levio principal;
- `levio_principals.principal_id` is the canonical owner anchor;
- Supabase `auth.users.id` or any provider subject remains only a provider
  reference;
- client-supplied `principal_id`, `owner_id`, provider reference, email, record
  ID, or subscription state is rejected as ownership proof;
- every returned resource belongs to the canonical principal;
- every parent context record belongs to the canonical principal;
- cross-principal ambiguity fails closed;
- missing or disabled workflow dependencies fail closed.

## Principal Resolution Checks Required Before Exposure

Future API implementation must verify:

- the auth context is authenticated;
- the session is active;
- the provider reference is present and valid;
- the provider reference status is active;
- the principal type is `registered_user`;
- the Levio principal status is active;
- the principal deletion state is active;
- the provider reference belongs to the authenticated session;
- no duplicate, missing, disabled, deleted, or ambiguous principal is accepted.

## Rollback Requirements Before Exposure

Before any future endpoint is exposed, the implementation must provide:

- deny-by-default route configuration or feature flag;
- fail-closed behavior when the route is disabled;
- response contracts that can be versioned or disabled without changing
  persistence data;
- no irreversible data write in the first route foundation;
- no hard deletion on rollback;
- no generated export file that becomes orphaned on rollback;
- no public UI promise that exceeds backend capability;
- logs that avoid protected payloads;
- status visibility that can remain available if request creation is disabled;
- separate schema rollback planning if later stages add durable request records.

## Product Integration Boundary

Future exposure must preserve:

- User Data Controls operate around decision simulation artifacts only;
- persistence remains storage for decision simulation artifacts, not chat history;
- API responses must not become AI answer surfaces;
- API routes must not call OpenAI or any AI Provider;
- API routes must not introduce billing or subscription entitlement behavior;
- API routes must not become generic assistant memory controls;
- UI exposure remains blocked until a separate product/UI scope is approved.

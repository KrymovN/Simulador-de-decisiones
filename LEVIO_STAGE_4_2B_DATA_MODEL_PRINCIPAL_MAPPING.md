This document defines the Stage 4.2B data model and principal mapping specification. It does not create runtime code, SQL, migrations, Supabase tables, storage buckets, API routes, auth runtime changes, simulator changes, dashboard changes, AI integration, memory, subscriptions, billing, or payments.

## 1. Purpose

Stage 4.2B converts the Stage 4.2A persistence architecture into a precise conceptual data model and principal mapping contract.

The goal is to define how Levio will later persist simulations, drafts, and history under stable owner principals without letting Supabase Auth, browser state, localStorage, dashboard UI, or simulator output accidentally define production ownership.

Stage 4.2B answers:

- what `principal_id` means;
- how `principal_id` relates to Supabase `auth.users.id`;
- which entities exist in the persistence model;
- who owns each entity;
- which operations are allowed per entity;
- which parts of future `SimulationResponse V2` may be saved;
- how guest-to-authenticated transitions remain bounded;
- how export, deletion, retention, consent, and auditability are preserved.

## 2. Principal Mapping Model

### 2.1 Canonical Levio Principal ID

`principal_id` is the canonical Levio-owned identifier for a durable owner.

Properties:

- generated by Levio, not by an auth provider;
- opaque and non-guessable;
- stable across email changes;
- stable across Supabase account recovery where the same person regains control of the same account;
- stable across future provider migration when identity continuity is proven;
- not derived from email, display name, subscription state, or provider access token;
- not exposed as an authorization claim that the client can choose.

Conceptual shape:

```text
principal_id
principal_type
principal_status
created_at
updated_at
verified_at
disabled_at
deletion_state
```

Initial durable `principal_type` for Stage 4.2 is:

```text
registered_user
```

Future principal types may include:

```text
guest_session
workspace
internal_operator
service
```

Only explicitly approved principal types may own user-owned persistence records.

### 2.2 Relation to Supabase `auth.users.id`

Supabase Auth is the approved authentication provider.

Supabase `auth.users.id` must be stored and treated as:

```text
provider = "supabase"
provider_reference = auth.users.id
provider_subject_type = "user"
```

It is not the canonical Levio owner ID for simulation persistence.

Required mapping direction:

```text
Supabase session
->
provider_reference
->
levio_principals.principal_id
->
owner-scoped persistence records
```

Runtime implications for future stages:

- authenticated requests resolve the Supabase provider reference server-side;
- persistence writes derive owner from the resolved Levio principal;
- client-supplied `principal_id`, `owner_id`, `provider_reference`, or email is untrusted;
- RLS planning must enforce the same mapping rather than relying on unscoped client claims.

### 2.3 Provider Reference

A provider reference links an external authentication identity to a Levio principal.

Conceptual fields:

```text
provider_reference_id
principal_id
provider
provider_subject
provider_subject_type
provider_email_snapshot
provider_email_verified
linked_at
last_seen_at
revoked_at
link_status
```

Rules:

- provider references are identity links, not owners of user content;
- provider email snapshots are metadata and may change;
- provider reference revocation must not delete user-owned data by itself;
- one Levio principal may later support multiple provider references only after account-linking rules are approved;
- two provider references must not be merged into one principal without explicit verified account-linking logic.

### 2.4 Guest Principal

Guest persistence is not approved for the first Stage 4.2 implementation.

The data model remains compatible with a future `guest_session` principal, but guest records must not be created until a later stage explicitly approves guest persistence.

Future guest principal properties:

- opaque identifier;
- short expiry;
- limited allowed record types;
- no durable memory;
- no subscription ownership;
- no cross-device recovery unless possession of the guest session is proven;
- claimable only through explicit guest-to-authenticated transition.

Until guest persistence is approved:

- anonymous and guest-like local data remains local/session scoped;
- localStorage saved demo simulations are not production records;
- Supabase anonymous sign-ins remain disabled by default.

### 2.5 Authenticated Principal

An authenticated principal is a `registered_user` Levio principal resolved from a valid Stage 4.1 Supabase Auth session.

Properties:

- may own durable simulations, drafts, and history entries;
- may request export/delete actions in future controls;
- may update only records owned by the same principal;
- does not automatically consent to memory, analytics, AI training, marketing, or workspace sharing.

Authenticated state is necessary but not sufficient for every action. Export, deletion, account linking, and future billing controls may require recent auth or step-up verification.

### 2.6 Account Transition

Account transition means moving eligible temporary data into a registered user's durable owner scope.

Supported conceptual transitions:

```text
anonymous_local_item -> registered_user
guest_session_item -> registered_user
provider_reference -> existing_principal
provider_reference -> new_principal
```

Rules:

- transition must be explicit, visible, and user-confirmed;
- source control and target account control must both be proven;
- transition must use idempotency;
- ineligible, expired, sensitive, hidden, or operational data must not be transferred;
- transition must not create memory consent;
- transition must not overwrite existing registered-user data silently;
- transition must not use subscription status as ownership proof.

### 2.7 Future Provider Migration Compatibility

The model must allow future migration away from Supabase Auth.

Migration compatibility requirements:

- user-owned records reference `principal_id`, not provider-native IDs;
- provider references are isolated from simulation payloads;
- product records do not store provider tokens;
- principal records can receive a new provider reference after verified migration;
- export can include provider-independent principal and ownership metadata without exposing secrets;
- deletion lifecycle can remove or detach provider references without orphaning owned records;
- provider migration does not imply data merge.

## 3. Ownership Rules

### 3.1 Cross-Entity Invariants

All Stage 4.2 persistence entities must obey these invariants:

- deny access when owner cannot be proven;
- resolve owner server-side from the authenticated session;
- never trust client-provided owner identifiers;
- record IDs alone never authorize access;
- list operations are owner-scoped;
- writes derive owner from the resolved Levio principal;
- deletion is a lifecycle state, not only a UI hide;
- export remains available regardless of subscription tier;
- saved simulation data does not become memory unless a later memory stage obtains explicit consent.

### 3.2 `levio_principals`

Owner:

- system-owned identity mapping record for a Levio principal;
- not user-owned decision content, but user-related personal data.

Access boundaries:

- readable by server-side auth/persistence code for the current authenticated principal;
- not directly listable by users across principals;
- not writable from client input.

Create permissions:

- created only through approved server-side account initialization or future guest-principal creation.

Read permissions:

- current principal may read a minimized self representation in future account/profile controls;
- server may read for authorization.

Update permissions:

- server-only updates for status, verification, provider-link changes, deletion state, and recovery state.

Delete permissions:

- account deletion triggers restriction, deletion, or minimal retained state according to approved deletion lifecycle.

Export eligibility:

- export may include principal ID, principal type, status, creation timestamp, and provider-independent account metadata;
- export must not include secrets, provider tokens, or internal security signals.

### 3.3 `simulation_records`

Owner:

- `registered_user` principal in the initial implementation.

Access boundaries:

- owner-only for read, update, archive, delete, and export;
- no anonymous or unrelated guest access;
- internal operators have no default access.

Create permissions:

- authenticated registered user only;
- requires explicit save or approved account-level save behavior;
- owner derived server-side from the session-to-principal mapping.

Read permissions:

- owner-only;
- list queries must scope by resolved owner;
- deleted/restricted records are excluded except for approved deletion/export workflows.

Update permissions:

- owner-only;
- permitted updates may include title, archive state, user-visible notes, later outcomes, or revision metadata where approved;
- regeneration creates a revision or history entry rather than silently replacing history.

Delete permissions:

- owner-only;
- deletion updates lifecycle state and cascades or restricts dependent history/drafts according to deletion rules.

Export eligibility:

- export eligible by default while active, archived, or deletion-pending;
- export includes user input, saved deterministic output, visible metadata, timestamps, provenance, and versions.

### 3.4 `simulation_drafts`

Owner:

- `registered_user` principal when server-side drafts are approved;
- anonymous local drafts remain outside production persistence;
- guest drafts are deferred until guest persistence is approved.

Access boundaries:

- owner-only for registered drafts;
- no cross-user draft reads;
- drafts must not be exposed through public simulator endpoints.

Create permissions:

- authenticated registered user for production server drafts;
- local anonymous draft creation remains browser-local and is not a production persistence write.

Read permissions:

- owner-only;
- expired drafts are inaccessible except for minimal operational deletion evidence.

Update permissions:

- owner-only;
- autosave must be visible and reversible if implemented;
- partial sensitive text must not be promoted into history or memory.

Delete permissions:

- owner-only;
- clear/discard actions delete or schedule deletion of active and temporary copies.

Export eligibility:

- saved registered drafts are export eligible;
- expired temporary drafts may be excluded if no longer retained;
- exported drafts preserve original language and timestamps.

### 3.5 `simulation_history_entries`

Owner:

- same `principal_id` as the parent `simulation_record`.

Access boundaries:

- owner-only;
- not a global activity log;
- operational logs must not be exposed as history.

Create permissions:

- server-created as part of approved simulation lifecycle events;
- not directly created by arbitrary client requests.

Read permissions:

- owner-only through parent simulation or owner-scoped history listing.

Update permissions:

- generally append-only;
- corrections create follow-up entries rather than rewriting material history.

Delete permissions:

- follows parent simulation deletion unless minimal legal/operational proof must be retained without source content.

Export eligibility:

- user-visible history entries are export eligible;
- export preserves event order, timestamps, event type, and relationships.

## 4. Entities Specification

This section is a conceptual resource specification. It is not SQL and must not be treated as a migration file.

### 4.1 `levio_principals`

Purpose:

- provide the stable Levio owner principal independent from auth provider IDs;
- support account recovery, provider migration, future provider linking, and future export/delete workflows.

Required fields:

```text
principal_id
principal_type
principal_status
primary_provider
primary_provider_reference
created_at
updated_at
verified_at
deletion_state
retention_rule
```

Optional fields:

```text
disabled_at
recovery_state
last_authenticated_at
last_provider_sync_at
locale_preference_reference
metadata_version
```

Lifecycle:

- created when a registered user principal is first initialized;
- updated when provider reference status changes;
- restricted or deleted during account deletion lifecycle;
- retained only minimally where operational/legal proof is required.

Relation to principal:

- this record is the principal anchor.

Privacy classification:

- user-related identity metadata;
- not decision content;
- may still be personal data.

Deletion behavior:

- account deletion must revoke access and delete, restrict, anonymize, or minimally retain according to legal/operational requirements;
- retained records must not recreate a hidden profile.

Export behavior:

- include minimal provider-independent account/principal metadata;
- exclude provider tokens, secrets, internal risk signals, and raw security logs.

### 4.2 `simulation_records`

Purpose:

- store an explicitly saved user-owned simulation result and its provenance.

Required fields:

```text
simulation_id
owner_principal_id
owner_principal_type
record_status
source_type
user_input_snapshot
deterministic_output_snapshot
simulation_response_version
decision_contract_version
language
safety_classification
recommendation_state
created_at
updated_at
deletion_state
retention_rule
export_eligible
```

Optional fields:

```text
title
user_note
clarification_snapshot
decision_model_snapshot
confidence_summary
archived_at
deleted_at
last_exported_at
originating_draft_id
parent_simulation_id
revision_label
```

Lifecycle:

- created by explicit save;
- updated only by owner-scoped operations;
- archived without deletion;
- deleted through lifecycle state;
- exported as part of user data export.

Relation to principal:

- belongs to `owner_principal_id`;
- may reference `levio_principals.principal_id` conceptually in a future schema;
- does not reference Supabase `auth.users.id` as owner.

Privacy classification:

- user-owned decision content;
- may contain sensitive personal, financial, employment, relationship, or health-adjacent context depending on user input;
- requires strong owner isolation.

Deletion behavior:

- deleting the parent simulation must address dependent history, revisions, derived summaries, and future memory links;
- operational proof may retain opaque IDs and deletion status only.

Export behavior:

- export includes source input, saved deterministic output, visible safety notices, timestamps, provenance, language, and contract versions;
- export distinguishes user-provided facts from system-derived analysis.

### 4.3 `simulation_drafts`

Purpose:

- store user-authored decision input before final save where server-side drafts are approved.

Required fields:

```text
draft_id
owner_principal_id
owner_principal_type
draft_status
draft_text_snapshot
language
created_at
updated_at
expires_at
deletion_state
retention_rule
export_eligible
```

Optional fields:

```text
clarification_answers_snapshot
structured_context_snapshot
autosave_enabled
last_autosaved_at
originating_surface
converted_simulation_id
discarded_at
```

Lifecycle:

- created when an authenticated user intentionally saves or uses approved visible autosave;
- updated by owner only;
- converted into a simulation record only through explicit save;
- expired, discarded, or deleted according to retention.

Relation to principal:

- belongs to `owner_principal_id`;
- future guest drafts must use a guest principal only after explicit approval.

Privacy classification:

- user-authored decision content;
- potentially sensitive even when incomplete.

Deletion behavior:

- discard clears active content and schedules deletion of retained temporary copies;
- expired drafts become inaccessible.

Export behavior:

- saved registered drafts are export eligible;
- expired or deleted temporary drafts are not exported unless retained under an approved recovery window.

### 4.4 `simulation_history_entries`

Purpose:

- preserve user-visible lifecycle, revision, and outcome events for saved simulations.

Required fields:

```text
history_entry_id
owner_principal_id
owner_principal_type
simulation_id
event_type
event_timestamp
event_source
user_visible
deletion_state
retention_rule
export_eligible
```

Optional fields:

```text
event_summary
before_reference
after_reference
revision_reference
outcome_snapshot
claim_transaction_reference
export_reference
```

Lifecycle:

- created by server-side lifecycle events;
- usually append-only;
- archived/deleted with parent simulation;
- retained minimally only for approved operational or legal reasons.

Relation to principal:

- same owner as parent simulation;
- parent mismatch is invalid.

Privacy classification:

- user-owned history when user-visible;
- operational metadata when internal-only, and must not contain raw decision content unless necessary.

Deletion behavior:

- follows parent simulation deletion;
- retained deletion evidence must be opaque and content-minimized.

Export behavior:

- user-visible entries are export eligible;
- internal operational entries are excluded unless needed for user-visible account activity.

## 5. Simulation Record Payload Policy

### 5.1 Saveable Parts of `SimulationResponse V2`

Future saved simulation records may persist only user-visible and user-controlled parts required for the saved decision record.

Potentially saveable:

- original user input snapshot;
- explicit clarification answers selected for the decision record;
- normalized decision summary;
- options, constraints, goals, assumptions, and known gaps;
- scenarios and scenario summaries;
- risk assessments;
- benefits and tradeoffs;
- recommendation category and rationale;
- withheld-recommendation or refusal state;
- confidence and completeness explanations;
- user-visible safety notices;
- lifecycle status;
- contract version;
- deterministic pipeline version;
- timestamps and language;
- provenance linking output to source input and engine version.

### 5.2 Parts That Must Not Be Saved by Default

Do not save by default:

- provider tokens or auth secrets;
- raw AI provider prompts or responses;
- hidden chain-of-thought or internal reasoning traces;
- raw operational logs;
- raw microphone audio;
- voice transcripts beyond active use unless the user explicitly saves text;
- temporary retry state;
- abuse-prevention signals that include sensitive content;
- unrelated analytics;
- embeddings or vectors;
- long-term memory records;
- cross-decision inferred profiles;
- unsupported inferences represented as facts;
- raw documents unless a separate document-storage stage approves them.

### 5.3 Payload Separation

Saved simulation payloads must separate:

```text
user_input_snapshot
deterministic_output_snapshot
user_visible_metadata
safety_flags
system_operational_metadata
```

Separation rules:

- user input remains user-owned content;
- deterministic output is user-controlled derived content when saved;
- metadata must be content-minimized;
- safety flags must be visible or explainable where they affect product behavior;
- operational metadata must avoid raw decision content where an opaque reference is enough.

### 5.4 Avoiding AI Coupling

Stage 4.2B does not approve AI provider storage.

Rules:

- persistence schema must not require OpenAI, model-provider, vector, embedding, or prompt-storage fields;
- future AI output may be saved only after it is mapped into the approved `SimulationResponse V2` contract and user-visible saved record shape;
- model/provider details, if later needed, must remain provenance metadata and must not become ownership data;
- AI providers never own saved simulations.

### 5.5 Avoiding Memory Coupling

Saved simulations must not automatically create memory.

Rules:

- saving a simulation is not consent to long-term memory;
- no memory table/entity is created in Stage 4.2B;
- memory promotion requires a later explicit user action and consent model;
- deleting a simulation must later delete or invalidate dependent memory unless an independent approved basis exists.

## 6. Guest-to-Authenticated Transition

### 6.1 Current Stage Decision

Guest persistence is deferred.

Stage 4.2B defines compatibility only. It does not approve guest tables, guest records, claim tokens, import APIs, or UI.

### 6.2 What Happens to Guest Data

Until guest persistence is approved:

- guest-like data remains local/session scoped;
- temporary simulator outputs are not production saved simulations;
- local demo saved simulations remain local demo data;
- clearing browser state may remove local data and Levio must not promise recovery.

If guest persistence is later approved:

- guest records must expire;
- expired guest records become inaccessible;
- unclaimed guest data is deleted or anonymized according to retention;
- guest ownership remains separate from registered-user ownership.

### 6.3 What May Transfer Later

Future explicit claim may transfer:

- user-authored draft text;
- explicitly saved guest simulation result;
- user-selected local demo simulation import;
- selected low-risk preferences such as language.

Every transfer must:

- prove source control;
- prove target registered account control;
- show eligible records;
- require explicit confirmation;
- be idempotent;
- record minimal provenance.

### 6.4 What Must Not Transfer

Do not transfer:

- raw processing traces;
- security logs;
- expired clarification context;
- raw microphone audio;
- hidden AI/provider traces;
- inferred memory;
- cross-decision profiles;
- another guest's data;
- another device's data without proof of control;
- subscription or billing state;
- unsupported sensitive data retained for undefined future use.

### 6.5 Consent Requirements

Claiming a draft or simulation is not memory consent.

Consent may be required later for:

- persistent sensitive decision history where consent is the chosen lawful basis;
- long-term memory promotion;
- cross-decision reuse;
- analytics or model-improvement use;
- workspace sharing.

### 6.6 Deferred to User Data Controls

The following remain deferred:

- import UI;
- guest claim UI;
- deletion controls;
- export controls;
- consent controls;
- memory promotion controls;
- workspace transfer controls;
- account deletion orchestration.

## 7. GDPR and Data Rights Compatibility

### 7.1 Export

Export compatibility requirements:

- every export-eligible record has stable identifiers;
- relationships between simulations, drafts, history, and principals are preserved;
- user-authored text keeps original language;
- user facts and deterministic inferences remain distinguishable;
- contract versions and timestamps are included;
- export excludes secrets, provider tokens, internal security signals, and another user's records;
- subscription downgrade or cancellation must not block export.

### 7.2 Deletion

Deletion compatibility requirements:

- every entity has a deletion state;
- deletion scopes are clear per record type;
- deleting a simulation addresses dependent history and future derived records;
- deleting a principal triggers owned-record deletion/restriction/anonymization according to lifecycle;
- operational proof keeps only opaque identifiers and status when required;
- legal exceptions are narrow, documented, and excluded from normal product processing.

### 7.3 Retention

Retention compatibility requirements:

- no indefinite silent default;
- retention is defined by category and purpose;
- drafts use short retention unless explicitly saved;
- guest records expire if later approved;
- saved simulations persist until deletion, account deletion, or configured retention;
- backups must follow an approved deletion propagation lifecycle in later implementation.

### 7.4 Data Minimization

Data minimization requirements:

- store only fields required for saved simulation, draft, history, ownership, export, deletion, or security purpose;
- keep auth provider references separate from simulation content;
- avoid raw decision content in operational logs;
- avoid hidden profiles;
- do not store AI prompts, vectors, embeddings, raw audio, or provider secrets in Stage 4.2 entities.

### 7.5 Consent Boundaries

Consent boundaries:

- registration is not consent;
- saving a simulation is not memory consent;
- saving a simulation is not analytics consent;
- saving a simulation is not AI training consent;
- consent for one purpose does not imply another;
- consent records are not implemented in Stage 4.2B but the model must remain compatible.

### 7.6 Auditability

Auditability requirements:

- ownership resolution must be traceable without exposing unnecessary decision content;
- create/update/delete/export events should have minimal operational evidence;
- guest/account claims require terminal states and idempotency evidence if later approved;
- internal operator access must be exceptional, least-privileged, and auditable in a future admin architecture;
- audit records must not become hidden decision history.

## 8. Stage 4.2C Prerequisites

Stage 4.2C may begin only after Stage 4.2B is accepted.

Required before schema planning:

1. Confirm `registered_user` is the only initial durable owner type.
2. Confirm guest persistence remains deferred or explicitly approve a separate guest schema scope.
3. Confirm `levio_principals` is the canonical principal mapping resource.
4. Confirm Supabase `auth.users.id` remains `provider_reference`, not owner ID.
5. Confirm exact schema target and Supabase project/database boundary.
6. Confirm data region and legal/privacy posture for real user data.
7. Confirm retention defaults for simulations, drafts, and history.
8. Confirm deletion states and cascade/restriction rules.
9. Confirm export fields and portable export shape.
10. Confirm RLS policy intent in prose before SQL exists.
11. Confirm server-side authorization test matrix.
12. Confirm rollback plan for schema planning and future migrations.
13. Confirm no subscriptions, AI, memory, analytics, vector storage, or billing are included.

Stage 4.2C must still not apply migrations, create tables, or write runtime code unless the owner explicitly approves a changed scope.
